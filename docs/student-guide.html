Student Hands-On Testing Guide

Everything you need to run BahaAI on your own laptop, click through every feature, and understand what each button and input field does. No coding required.

Runs 100% on your computer Docker ~10 minutes to first flood alert
What is BahaAI? A flood early-warning system for Consolacion, Cebu. It watches water levels from sensors, warns people before a flood gets dangerous, and predicts how soon an area will flood. The sensor hardware is simulated in software — you'll trigger "floods" yourself with a command and watch the whole system react.

1. Start the App (Docker)

You only need Docker Desktop installed — nothing else (no PHP, no database).

  1. Install Docker Desktop from docker.com. On Windows, restart your computer after installing. Open Docker Desktop and wait until it says "Engine running".
  2. Get the project. Go to github.com/Jumbe01/baha-ai, click the green Code button → Download ZIP, and extract it (e.g. to your Desktop).
  3. Open a terminal in that folder (the folder that has docker-compose.yml).
    • Windows: open the folder in File Explorer, click the address bar, type cmd, press Enter.
    • macOS: right-click the folder → New Terminal at Folder.
  4. Run:
    docker compose up --build
    The first time takes a few minutes. Wait until you see:
    BahaAI is running  ->  http://localhost:8000
  5. Open your browser to http://localhost:8000.
Keep this terminal open while you use the app — it's running the server. To stop the app later, press Ctrl+C in this terminal.

3. Command Cheat Sheet (Docker)

These commands make the system "do things" — like simulating a storm. Run them in a second terminal, opened in the same project folder, while the app is running.

Important: run commands INSIDE Docker The app's database lives inside Docker. If you run php artisan ... directly in VS Code, it talks to a different database and nothing will change on the website. Always prefix commands with docker compose exec app so they hit the same database the website uses.
What it doesCommand
Trigger a storm — pushes water levels up and creates warning/critical alerts docker compose exec app php artisan sensors:simulate --storm
Add normal (calm) sensor readings docker compose exec app php artisan sensors:simulate
Generate flood predictions for all sensors docker compose exec app php artisan predictions:generate
Refresh weather now docker compose exec app php artisan weather:fetch
Reset all demo data to a fresh state docker compose exec app php artisan migrate:fresh --seed

After running any command, go back to your browser and refresh the page to see the effect.

4. Try It Out — "Make a Flood Happen"

This 5-minute walkthrough takes you through the core of the whole system. Do it logged in as the admin.

  1. Look at the calm state. Open Dashboard. Note the "Risk Status" (probably Clear or a few warnings) and the 24-hour water-level chart.
  2. Cause a storm. In your second terminal, run: 
    docker compose exec app php artisan sensors:simulate --storm
    You'll see a list of sensors turning WARNING and CRITICAL.
  3. See the alerts. In the browser, open Alerts (refresh if needed). New alerts appear, color-coded by severity. Click one to see its details and the list of who was notified.
  4. Check the dashboard again. The Risk Status now shows red/yellow counts, and flood zones on the right show their status.
  5. Generate predictions. Run: 
    docker compose exec app php artisan predictions:generate
    Then open Predictions. Each sensor now shows a forecast: how fast water is rising, how many minutes until critical, and a confidence score.
  6. Open the Flood Map. Zones are shaded green/yellow/red by risk. Click a sensor dot to see its reading.
  7. Resolve an alert. Back on Alerts, open a critical one and click Resolve — it moves from active to resolved.
  8. Reset when done. To go back to a clean slate: 
    docker compose exec app php artisan migrate:fresh --seed
That's the whole loop! Sensor reading → risk check → alert → notification → prediction → map. Everything else in the app supports or configures this loop.

Feature & Input-Field Reference

What each screen does, and exactly what to type in every input field. (Admin/Staff only screens are marked.)

Flood Zones Admin

What it does: defines the flood-prone areas and the water heights that count as safe, warning, or dangerous. These numbers control every alert and prediction.

FieldWhat to enter
Zone NameAny name, e.g. Tayud River Basin. Required.
DescriptionOptional notes about the area.
BarangayPick from the dropdown. This decides which residents get alerts for this zone.
Safe (m)Water height that is normal, usually 0.
Warning (m)Height where you start warning people, e.g. 1.5. Must be higher than Safe.
Critical (m)Dangerous height, e.g. 3.0. Must be higher than Warning.
Rule: the three numbers must increase — Safe < Warning < Critical — or the form won't save.

Sensors Admin

What it does: registers a monitoring device inside a flood zone. Only active sensors are shown on the dashboard and map.

FieldWhat to enter
Sensor NameAny label, e.g. Sensor-015. Required.
Flood ZonePick which zone this sensor belongs to.
Sensor TypeUltrasonic (water level), Rain Gauge, or Pressure.
Latitude / LongitudeMap position. Defaults are near Consolacion (~10.3667, 123.9567) — you can leave them.
Status (edit only)Active, Inactive, or Maintenance.

Alerts

What it does: shows every flood warning. Most alerts are created automatically when water crosses a threshold. Staff can also raise one by hand.

Viewing: use the filters at the top (status, severity, zone, or search) to narrow the list. Click any row to open the full detail, including the recipient and notification log.

Creating a manual alert Staff/Admin — the Create Alert form:

FieldWhat to enter
Flood ZoneWhich area the alert is for. Residents of its barangay get notified.
SeverityWarning or Critical.
TitleShort headline, e.g. Flash flood reported near market.
MessageThe details residents will see.

Clicking Create & Dispatch immediately sends (simulated) notifications. Open the new alert to see the delivery log.

Predictions

What it does: forecasts how soon each sensor will reach a dangerous level, using the trend of recent readings. No input fields — just press Generate Predictions (Staff/Admin) to refresh.

What you seeMeaning
Current / Predicted levelWater now vs. estimated in 1 hour.
Time to criticalMinutes until the danger level, or "Stable" if not rising.
ConfidenceHow trustworthy the forecast is (higher = steadier trend).
RecommendationSuggested action based on the risk.
Tip: if predictions look empty, run a storm first, then predictions:generate — a sensor needs a few recent readings to forecast.

Actuation Staff/Admin

What it does: a control panel for flood devices (pumps, sirens, floodgates). Simulated — clicking a control just records the action.

  • Auto / Manual buttons — switch a device's control mode.
  • Turn On / Turn Off — only available in Manual mode. Every action is written to the activity log below.

Analytics Staff/Admin

Does SMS and Email Actually Work?

Short answer: No real text messages or emails are sent — and that's intentional for this project.

Alert notifications (SMS / Email / Push)

When an alert fires, BahaAI figures out who should be notified and records a notification for each person and channel — but it does not connect to a real phone network or email server. This is a safe, self-contained simulation so the system can be demonstrated without sending real messages or needing paid services.

Where to see it: open any alert (Alerts → click a row) and scroll to the Notification Log. You'll see each recipient, the channel (SMS/email/push), the phone/email it "went to", and the time. That proves the notification logic works end-to-end.

The sign-up verification email (OTP)

Registration generates a 6-digit code and "emails" it — but the demo is set to write emails to a log file instead of delivering them, so you won't receive it in an inbox.

Because of this, the simplest way to explore every role is to use the built-in admin@bahaai.test account and create other accounts from the Users page (those need no code). If you really want to finish a self-registration, you can look up the latest code with:

docker compose exec app php artisan tinker --execute="echo App\Models\OtpCode::latest('id')->first()->code;"
Could it send real messages? Yes — the system was built so that a real SMS gateway (e.g. Twilio or Semaphore) and a real email service can be plugged in later by changing one settings area, without rewriting the app. For classroom testing, the simulation shows exactly the same behavior.

Reset & Troubleshoot

Start over with fresh data

docker compose exec app php artisan migrate:fresh --seed

This wipes everything and reloads the original demo data (zones, sensors, 30 days of readings, sample incidents). The admin login stays admin@bahaai.test / password.

Completely reset the containers

# stops everything and deletes the database volume
docker compose down -v
docker compose up --build
ProblemFix
Browser says the page can't be reachedMake sure the docker compose up terminal is still running and shows "BahaAI is running".
port is already allocated / 8000 in useAnother app uses port 8000. Copy .env.docker.example to .env, set APP_PORT=8080, then open localhost:8080.
Cannot connect to the Docker daemonDocker Desktop isn't running. Open it, wait for "Engine running", try again.
A command "did nothing" on the websiteYou probably ran it without docker compose exec app in front. Re-run it with that prefix.
The first build seems stuckNormal — it downloads several hundred MB the first time. Give it a few minutes.
