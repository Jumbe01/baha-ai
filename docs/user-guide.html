BahaAI User Guide & System Manual

A complete guide to the BahaAI flood-monitoring platform — what each user can do, where the data comes from, and how the system turns sensor readings into predictions and alerts.

Municipality of Consolacion, Cebu Web Platform Capstone / Thesis Edition

1. System Overview

BahaAI is a web-based flood early-warning system. It continuously monitors water levels across flood-prone barangays, predicts when a location may reach a dangerous level, and automatically alerts the right people in time to act.

The platform is organized around four ideas:

  • Flood Zones — defined areas (river basins, creeks, lowlands) in Consolacion, each with its own safe / warning / critical water-level thresholds.
  • Sensors — devices placed inside flood zones that report water level, rainfall, temperature, and humidity.
  • Readings — the time-stamped measurements those sensors produce.
  • Alerts & Predictions — what the system generates when readings cross a threshold or trend toward one.
Important — about the hardware This is the web platform portion of the thesis. The physical IoT hardware (ultrasonic sensors, rain gauges, pumps, sirens) is simulated in software so the system can be demonstrated end-to-end without deploying real devices. Everything you see — water levels, alerts, predictions — runs on the same logic that real sensor data would flow through. See Where Data Comes From for the details.

3. Logging In & Account Verification

New residents register themselves; staff and admin accounts are created by an administrator.

  1. Go to the registration page and enter your name, email, mobile number, and barangay.
  2. The system emails you a 6-digit one-time password (OTP) that is valid for 10 minutes.
  3. Enter the code on the verification screen to activate your account.
  4. After verifying, you are taken to your dashboard.
Tip Admin-created accounts (staff or other admins) are pre-verified — they can log in immediately without the OTP step.

4. Where Does the Data Come From?

This is the most common question, so here is the full picture. Water-level data enters the system through three paths, all of which end up as sensor readings in the database:

① Historical seed data (30 days)

When the system is first set up, each sensor is given 30 days of back-history at 15-minute intervals. These values are generated mathematically to look like real river behavior:

// water level = base + daily tide + seasonal trend + noise + occasional storm water_level = base + 0.30 · sin(2π · hour / 24) // tidal day/night cycle + 0.50 · sin(2π · (day−150) / 365) // wet/dry season + random noise + storm surge // ~3% of readings spike sharply

This gives realistic-looking graphs with natural rises and falls, plus the occasional storm spike — exactly what you would see from a real river gauge.

② Live simulation command

For demonstrations, staff can run a command that injects a fresh reading into every active sensor right now:

  • sensors:simulate — generates normal (safe) readings.
  • sensors:simulate --storm — generates elevated readings to trigger warnings and critical alerts, so you can watch the whole alert pipeline fire live.

③ The sensor data API

Each reading can also be posted directly to an API endpoint:

POST /api/sensors/{sensor}/readings { "water_level": 2.4, "rainfall": 12.5, "temperature": 28, "humidity": 80 }

This is the path a real IoT device would use. Because the simulation and the real hardware both arrive here, the rest of the system — alerts, predictions, the map — behaves identically whether the data is simulated or real. Swapping in physical sensors later requires no changes to the web platform.

What each reading contains

water_levelHeight of water in metres — the primary value.
rainfallRainfall in millimetres.
temperatureAir temperature in °C.
humidityRelative humidity (%).
recorded_atTimestamp of the measurement.

5. Risk Levels & Thresholds

Every flood zone defines three water-level thresholds. The current water level is compared against them to produce a risk level:

RiskConditionMeaning
Safelevel < warning thresholdNormal conditions. Routine monitoring.
Warningwarning ≤ level < criticalWater is rising. Prepare and watch closely.
Criticallevel ≥ critical thresholdDangerous. Evacuation may be required.

Typical thresholds for a zone are 0 m (safe), 1.5 m (warning), and 3.0 m (critical), but each zone can be tuned individually by an administrator because every river and creek floods at a different height.

Color convention Across the whole platform, green = safe, yellow = warning, and red = critical. The dashboard, water-level gauges, map, and alerts all use these same colors.

6. How Predictions Are Created

This is the "AI" part of BahaAI. Instead of only reacting when water is already dangerous, the system forecasts how soon a sensor will reach the critical level, giving responders a head start.

The method: linear regression

For each sensor, the prediction engine looks at the last 2 hours (120 minutes) of readings and fits a straight line through them — a classic linear regression. The line tells us two things: how fast the water is rising, and how well that trend fits the data.

The calculations

// 1. Rate of rise — the slope of the fitted line (metres per minute) rate_of_rise = slope from linear regression of (time, water_level) // 2. Time until the water reaches the critical threshold minutes_to_critical = (critical_threshold − current_level) / rate_of_rise // 3. Confidence — how well the straight line fits the real readings (R²) confidence (%) = R² × 100 // 4. Predicted level one hour from now predicted_level = intercept + slope × (now + 60 min)

What each output means

OutputPlain-English meaning
Rate of riseHow quickly water is climbing. A larger number means faster rising and less time to react.
Time to criticalEstimated minutes until the zone hits its critical threshold. Shows "Stable" when the water is flat or falling.
Confidence0–100%. High confidence means the recent readings form a clean, steady trend, so the forecast is trustworthy. Low confidence means the data is noisy/erratic — treat the estimate with caution.
RecommendationAn automatic action message, e.g. "Immediate evacuation advised — critical level projected in ~25 minutes."

How a prediction is classified

  • Critical — water is already at critical, or it is projected to reach critical within 60 minutes.
  • Warning — current or predicted level is at or above the warning threshold.
  • Safe — stable conditions.

When predictions run

Predictions are regenerated automatically every 15 minutes by a scheduled task. Staff can also press "Generate Predictions" on the Predictions page to refresh them on demand. The page then shows a forecast chart for each sensor, with a dashed projection line and a red line marking the critical threshold.

Why a prediction might say "No prediction yet" A sensor needs at least 3 readings in the last 2 hours to fit a line. A brand-new sensor, or one whose recent data is older than 2 hours, will be skipped until enough fresh readings arrive.

Who gets notified

When an alert fires, the system sends it to:

  • Residents in the affected barangay — only people in the flooding area are bothered, so alerts stay relevant.
  • All staff and administrators — responders always see every alert, regardless of location.

How notifications are delivered

Each recipient is notified through the channels they enabled in their profile — SMS, email, and/or push. In this web edition, SMS/email/push delivery is simulated and recorded in a notification log (you can open any alert and see exactly who was contacted, on which channel, and when). Connecting a real SMS gateway later only means replacing that one logging step.

No duplicate spam The system will not create a second alert of the same severity for the same sensor while one is still active. You get one warning alert and, if the water keeps rising, one critical alert — not a flood of repeats.

Resolving alerts

Staff and admins can mark an alert as Resolved once the situation is handled. Residents see alerts but cannot resolve them. Staff can also raise a manual alert (e.g. based on a phone report from the field) that dispatches notifications the same way an automatic one does.

8. Weather Data

The Weather page shows current conditions and a forecast for Consolacion. Data comes from the OpenWeatherMap service (free tier).

  • If an OpenWeatherMap API key is configured, live weather is fetched and cached for 30 minutes to stay within the free quota.
  • If no key is set, the system falls back to realistic simulated weather so the page always works for a demo. A "Simulated Data" badge is shown in that case.

Weather is refreshed automatically every hour by a scheduled task.

9. The Flood Map

The map gives a geographic, at-a-glance view of the whole municipality. It is built with MapLibre over OpenStreetMap base tiles and shows:

  • Flood-zone polygons — each area shaded green / yellow / red by its current risk level.
  • Sensor markers — click any sensor to see a popup with its name, zone, current water level, and risk badge.

The separate GPS Alerts page focuses on emergencies: it highlights only the zones that are currently flooding and draws an evacuation route from each one to the designated evacuation center.

11. Role Guide: DRRMO / Staff

📋

DRRMO / Barangay Staff

The operational responders who monitor conditions and act on alerts.

Your menu

Dashboard · Water Levels · Alerts · Predictions · Weather · Flood Map · GPS Alerts · Analytics · Actuation

Day-to-day monitoring

  • Dashboard — your control center: active sensor count, current risk summary, the highest water level right now, and a 24-hour trend chart.
  • Water Levels — a card per sensor with a live gauge; click one for its full history and charts.
  • Predictions — see which sensors are trending toward critical and how soon; press Generate to refresh.

Handling alerts

  • Review incoming alerts, filter them by status / severity / zone, and open one to see its full recipient and delivery log.
  • Resolve an alert when the situation is over.
  • Create a manual alert when you have field information the sensors haven't captured — it notifies residents and staff just like an automatic alert.

Smart Actuation

The Actuation page is a control panel for flood-mitigation devices (pumps, sirens, floodgates, valves). For each device you can:

  • Switch between Auto and Manual mode.
  • In manual mode, turn the device on or off. Every action is written to the activity log with your name and a timestamp.

Analytics & reports

  • Filter historical data by date range and review daily water-level trends and incidents-by-zone charts.
  • Export the flood-incident history as CSV (for spreadsheets) or PDF (for formal reports).

14. Frequently Asked Questions

Is the sensor data real?

In this thesis edition it is simulated, but it flows through the exact same logic real data would. Real sensors can be connected later via the API with no changes to the platform.

How accurate are the predictions?

Each prediction comes with a confidence score. When water rises in a steady trend, confidence is high and the time-to-critical estimate is reliable. When readings are erratic, confidence drops — a built-in honesty signal telling responders how much to trust the forecast.

Why didn't I get an alert?

Alerts go to residents in the affected barangay. Check that your barangay is correct in your profile, that your notification channels are enabled, and that the event actually crossed the warning/critical threshold for your zone.

Can I see who an alert was sent to?

Yes — staff and admins can open any alert to view the complete list of recipients and a delivery log showing each channel (SMS / email / push) and timestamp.

What happens when water keeps rising after a warning?

If it crosses the critical threshold, a separate critical alert is created and dispatched. You won't get repeated copies of the same warning, only the escalation.
